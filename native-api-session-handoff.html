1 · Architecture — the declaration-driven static registry

Every Native-API/Hook op is one HookOp enum case whose properties declare its arguments, return shape, portability, and version floor. The inspector UI is generated from those declarations + the APICatalog mapping — there is almost no per-op UI code. A graph run dispatches each op through HookEngine.apply, shared by the Graph executor and the headless Lab runners.

Adding / changing an op = the 4-touch-point recipe

The CLAUDE.md recipe said three; it is really four (the 4th is hardcoded and the build can't catch a miss):

  1. Core/HookDef.swift — add the HookOp case and every exhaustive computed-property switch: displayName, detail, paramKeys, returnShape, outputPreview, defaultOutputVar. (phase defaults .both, portability defaults .universal — only override when needed.) New params → add to HookParam (label + placeholder; control defaults to .text).
  2. Core/APICatalog.swift — add an APICatalogEntry to the available[] array: official symbol · signature · per-arg APIArgument (link each to its param: or fromInput:) · docPath · keywords. Version-gated ops set since/fallback.
  3. Core/HookEngine.swift — add the apply() branch returning a typed HookValue (.text/.list/.number/.object). Implement it: NL logic → add to LanguageTools; pure compute → a private func; heavy / I-O work → nonisolated async off the main actor (see Vision + script).
  4. Views/NodeInspector.swift — add the op to the hardcoded per-node-kind candidates list (.nativeAPI = NL/FM/Vision ops · .hook = glue/script ops). the compiler cannot catch a missing entry — the op just silently won't appear in the picker

Version-gate (structured, not prose)

APICatalogEntry gained since / fallback / computed availabilityNote. countTokens was the live over-claim — listed .available while the executor runs a heuristic — now declares since:"26.4", fallback:"TokenEstimator heuristic"; OpRow renders a macOS 26.4+ badge. One version-annotated registry, not one-per-OS: rows carry the floor, the executor does the #available dispatch.

os_signpost

GraphExecutor.signposter (OSSignposter on an OSLog with category .pointsOfInterest) wraps each node in the run loop with one interval labelled "<kind> · <apiName>" + an ok/error end. No-op when not recording; identical/harmless on iOS. Lets a user profile a graph run in Instruments.

6 · Landmines (each cost time this session)

docs/ is git-ignored. The PRD §5.4.1 op-spec edits this session live on disk only, never committed. Don't assume git history shows the spec.
candidates in NodeInspector is hardcoded per node-kind, NOT catalog-derived. Add a HookOp + catalog entry and the op still won't appear in the picker until you add it here. The build won't tell you.
Exhaustive-switch trap. Editing a HookOp computed-prop switch, it's easy to drop an existing case (I dropped countTokens from outputPreview). The build catches it because the switch is exhaustive — but the combined nil case can hide the omission. Re-check after each edit.
countTokens is SDK-gated. SystemLanguageModel.tokenCount(for:) / contextSize (macOS 26.4) are absent in this 26.2 SDK → heuristic TokenEstimator fallback. This is the canonical version-gated op.
The new Swift Vision API IS in this SDK. RecognizeTextRequest / DetectBarcodesRequest / CSUserQuery all compile here (probe-confirmed) — that's why OCR/barcode could be promoted. Always probe SDK presence before promoting a .planned op.
