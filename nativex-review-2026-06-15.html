Verdict

Strong, coherent product. One headline overclaim. A handful of fixable rough edges.

This is a real, well-architected workbench — not a demo. It runs live on-device LanguageModelSession calls with the best run-feedback I've seen in a node editor, a genuinely novel API-teaching inspector, and a working LLM-as-judge. The one substantive pitch-vs-product gap is cross-provider benchmarking, which the spec sells as current but the code does not implement. Notably, two items from my first-pass notes turned out to be already solved on close inspection — I've corrected them rather than ship a wrong critique. Net: keep the engineering, reconcile the one claim, and the UX bugs below are mostly closed already.

00  Method & integrity

I drove the Playground tab live — canvas, inspector, and a full on-device run (German-tutor graph) — and code-grounded the Datasets, Lab, Run History and Schema-editor tabs, which I couldn't reach via synthetic clicks (the window's unified title-bar swallows automated clicks in the tab strip — an automation ceiling, not an app defect).

Before publishing, I re-verified every claim against the current branch. That caught two first-pass errors (see Corrections) — exactly the kind of stale criticism that shouldn't reach a team doc. The findings below are what survived verification.

The API-teaching inspector

Op catalog grouped by framework, an iOS · macOS portability badge, and the title↔symbol duality (friendly name + exact Swift API). Nothing else in the FM-tooling space teaches the underlying API at the point of use like this.

Core/APICatalog.swift · Views/NodeInspector.swift

Live execution feedback

A top-center "running" pill, a cascade of green ✓ checks onto finished nodes, ambient canvas radiance, and the active FM node glowing. The canvas is the debugger — and it's polished.

Real on-device pipeline + judge

The executor streams from a live default-model session; the LLM-as-judge is a real guided-generation scorer with judge-vs-human agreement; reference eval is wired end-to-end.

02  Findings

F1 Cross-provider benchmarking is pitched as current, but absent in code High Confirmed

The PRD repeatedly sells routing one graph across on-device → Private Cloud Compute → Claude → Gemini "to benchmark cross-provider quality and latency from one canvas," as a present capability. The code does none of it.

Evidence
GenConfig — only on-device GenerationOptions fields; no backend/provider 3× LanguageModelSession sites → default on-device model GraphExecutor.swift:311 · Runners.swift:92 · Judge.swift:54 0 refs to Claude/Gemini/PCC/provider in Core·Views·Models·Engines PRD §1.2 · §4.3 · §5.2 · §5.5 — not in roadmap §8

Note: PRD §1.1's framework-level statement (Apple's LanguageModel protocol is open) is accurate background — the gap is the product-level claims that NativeX does this today.

Decision needed (product)

Either build a backend selector on the FM node — start with on-device ↔ PCC, both Apple-native and low-lift — or own the on-device-first scope cleanly. Building the full provider matrix is a feature, not a fix, so it's out of scope for this pass.

Shipped (conservative, reversible)

Reconciled the spec: a dated status note at the top of the PRD, an inline "(planned)" marker on the FM-node spec §5.2, and a new roadmap entry §8.7. This does not build the feature — it stops the doc from overclaiming until you decide.

F2 A run's result card can render off-screen and be unreachable Medium In progress

Live-observed: I ran a graph and could not see its output. The single-run result card is placed to the right of the terminal FM node — usually the right-most node — so it overflows past all content, and Fit-to-view framed only the nodes, not the card.

Evidence
card offset f.maxX + 28 → right of the terminal FM fitToView() unioned only node/group rects — GraphEngine.swift:540
Shipped

Introduced engine.resultCardRect as the single source of truth for both placement and framing, and unioned it into fitToView(). Fit-to-view (⌘0 / the canvas Fit button) now always frames the run output alongside the nodes — it can no longer strand off-edge.

F3 Inspector header: the editable title sits flush against the captions Medium In progress

The reported screenshot bug. In the node inspector header, the editable title TextField had only a 2pt gap above the kind / portability / API caption rows — an editable bordered field flush against text reads as broken.

Evidence
header() VStack spacing DS.Space.xxs = 2pt — NodeInspector.swift:75
Shipped (+ app-wide audit)

Title↔caption-group gap is now DS.Space.sm (8pt); caption lines stay grouped at xxs. Audited every xxs use app-wide: also relaxed the two Datasets sheet title/subtitle headers (xxs→xs). Deliberately left ~20 other xxs uses untouched — list rows, badges, hairlines, the DSField label/caption — those are intentional tight groupings, not the bug.

F4 No keyboard path to the four tabs; menu bar is the empty scaffold Low In progress

For a "Native-First" app, primary navigation had no keyboard shortcut and the macOS menu bar carried none of the app's commands. ContentView bound only ⌘\ (focus mode); the tabs were bare .tag(0–3).

Shipped (partial)

Added ⌘1–4 to jump straight to Playground · Datasets · Lab · Run History.

Deferred — why

Populating the menu bar (View/Edit CommandGroups) needs Scene-level wiring and a product call on menu structure — flagged rather than guessed.

F5 Canvas captions truncate the very API symbols they teach Low Recommendation

On the canvas, the API captions that are the product's core idea read DynamicGenerationS…, LanguageModelS…, even the title Instruc…. The full symbol shows in the inspector — but you spend your time on the canvas.

Recommendation

Widen the node header, or add a .help() tooltip with the full symbol. Low-risk, deferred to keep this pass surgical.

F6 Click ambiguity: select vs. inline rename; inspector off by default Low Recommendation

Clicking a node sometimes selects, sometimes drops into inline title-rename depending on prior selection. Separately, the inspector — the primary config surface — is off by default and discoverable only via ⌘⌥I.

Recommendation — deferred

Disambiguate rename (double-click / explicit affordance) and consider opening the inspector on node-select. Both are interaction-design changes with regression risk / opinionated UX — they want product input, not a unilateral edit.

03  Corrections — first-pass claims that didn't survive verification

retracted "Datasets can't store gold / reference outputs"

Incorrect. expectedOutputField exists and is fully wired (DatasetsView.swift:537 → loaded from ExampleModel.expectedOutput → scored by the Lab's reference evaluator in Pipeline.swift). Objective reference-based eval is usable through the app today. The only real defect was a stale file-header comment still calling it "reference-free / a roadmap item" — now fixed.

softened "Token accounting is oversold as a reliability feature"

Overstated. The PRD is explicitly version-gated and honest — exact counts via response.usage on 26.4+, a clearly-stated heuristic fallback (CJK ≈1 tok/char; Latin ≈1 tok/4 chars) on 26.0–26.3 — and the UI labels figures "~estimated." No change warranted. Residual nuance only: the current pre-26.4 build is heuristic-in-practice, which the UI already communicates.

05  Deferred, with reasons

  • Cross-provider feature build — a product decision + significant feature, not a fix. The doc is reconciled; the call is yours (F1).
  • Menu-bar Commands — needs Scene-level CommandGroup wiring and a structure decision (F4).
  • Inspector open-on-select — opinionated UX; default-off may be intentional.
  • Canvas caption truncation — low-risk polish, easy follow-up (F5).
  • Select-vs-rename disambiguation — interaction change with regression risk (F6).

Principle: fix the clear, low-risk, well-scoped issues surgically and commit each; flag the strategic or risky ones rather than ram them in.
