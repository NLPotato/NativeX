① On-device verification — do this first

Run destination My Mac, Apple Intelligence on, Siri language = device language. (Never add #Playground blocks — Xcode auto-runs them and crashes the model.)

  1. Import the test data. Datasets tab → the ↓ (Import) button (top-right, next to +) → pick /Users/lwj/Downloads/german-test.csv. Expect a “german-test” Custom dataset, 4 rows, columns id · input · proficiency · language.
  2. Insert the test graph. Graph tab → Load menu → “Insert example: compare × dataset”. You get Lane A (single-shot) + Lane B (extract → elaborate) + a Compare node.
  3. Rebind the Input to the dataset. Select the “Input (rebind to dataset)” node → inspector → Source = Dataset → pick german-test. The seeded edges map input/proficiency/language by name, so they keep resolving once columns match.
  4. Save the graph (gives it a name for the Lab labels).
  5. Run. Select the Compare node → the button now reads “Run comparison × german-test (4 rows)” → click it. That's 4 rows × 2 lanes = 8 generations, sequential & on-device (lane B does 2 calls/row, so ~12 model calls total — give it a minute).
  6. Verify the three surfaces:
    • ▸ Compare sheet: 2 lanes, “4 rows · N% decoded”, per-lane avg latency/tokens + a row-1 sample output.
    • ▸ Lab tab: one sweep, 2 experiments (“… · Lane A · single-shot”, “… · Lane B · elaborate”), 4 runs each. Lane B's p95 latency should be clearly higher (2 calls). Leaderboard ranks them.
    • ▸ Run History: 4 new entries (“<graph> · <row> · compare”), each showing FM A + FM B1 + FM B2 steps.
Quality comparison (the actual goal): there's no expected-output column, so the leaderboard ranks on decode/latency/tokens/language only — no accuracy score. Judge quality by eyeballing the per-row outputs in Run History (single-shot gloss vs. the 2-step elaboration).

② What Phase 3 does (mechanics)

All in Core/GraphCompareRunner.swift. One pass of the executor runs every FM in the graph; the runner just collects each lane's terminal output. Phase 2 (single input) and Phase 3 (dataset) are one code path — single input is “one pseudo-row (row = nil)”.

  • Fan-out: for each dataset row → GraphExecutor.run(graph, row:) → per lane, one RunModel. M lanes → M ExperimentModels (one each), N runs, all sharing one sweepID → the Lab's VariantStats leaderboard ranks them.
  • Honest cost: a lane's latencyMs + tokens sum its whole FM chain — chainFMIDs() walks backward from the terminal FM (edges + implicit member→group) collecting every upstream FM. So Lane B counts FM B1 + FM B2, not just the last call. Decode/quality is scored on the terminal output only.
  • Both output homes: the Lab sweep and one TraceModel per row → Run History (each trace covers all lanes' FMs).
  • Live columns: re-resolves the dataset Input's datasetColumns from the live dataset at run start (same fix as the batch lane), so a column added after binding isn't dropped.

③ Gotchas

Seeded Input is STATIC. If you skip step 3 (rebind to dataset), “Run comparison” just runs once on the static sentence (Phase 2) — not the 4 rows. The button label tells you which mode you're in.
Graph FM runs DYNAMIC schemas only. The “gloss schema” in the graph is SchemaDef.glossLike on a .guided block — not the typed Gloss.Result @Generable. There is no typed-@Generable path in GraphExecutor.runFM ([line 266]). If you ever need the typed schema in-graph, that's new work.
{{words}} = FM B1's full gloss JSON. Lane B's step-2 prompt receives FM B1's entire output (pretty gloss JSON), not a clean surface list. Fine for the model to read; for a tidy list, drop a JSON-extract hook between FM B1 and Current B2.
Pre-existing warning, not mine. GraphEngine.swift:106 “main actor-isolated initializer 'init()'…” is on the untouched init(graph: GraphDef = .init()) and is present at HEAD before this work. Leave it, or fix separately (e.g. init(graph: GraphDef? = nil)).
  • App is unsandboxed (ENABLE_APP_SANDBOX = NO) → fileImporter reads paths directly, no entitlement. The two startAccessingSecurityScopedResource() calls are harmless no-ops here.
  • Tokens are estimated (Apple exposes no token API): CJK ≈1 tok/char, else ≈1 tok/4 chars. German → the /4 branch.
  • A structural throw (validation/cycle) on row 1 cleans up the empty experiments and aborts; it's deterministic so it'd fail every row anyway.
